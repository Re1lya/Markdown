# 技术选型思路

## 1. 文档目的

本文用于沉淀平台工程 POC 的技术选型思路，说明每个技术组件在整体架构中的位置、解决的问题、为什么需要它，以及其他小组接入服务时应如何理解整条部署链路。

当前 POC 的核心目标不是单独部署一个应用，而是展示一套从服务注册、环境编排、CI/CD、GitOps 到开发者门户的端到端平台能力。

本文只记录当前 POC 中已经讨论和使用过的技术选型，不把尚未落地的生产级能力写成已完成能力。涉及公网域名、企业镜像仓库、正式权限体系、多租户审批等内容，均属于后续增强。

## 2. 总体架构思路

整体链路可以概括为：

```text
开发者 / 平台使用者
  -> Backstage Dev Portal
  -> 创建或查看服务
  -> Tekton 处理 CI 流程
  -> 镜像构建与推送
  -> GitOps 仓库更新
  -> Argo CD 同步集群资源
  -> Kubernetes 运行服务
  -> Envoy Gateway 暴露服务访问入口
  -> Backstage Catalog 回链到服务页面
```

平台底层能力由 Kubernetes 承载，Crossplane 用于平台资源抽象和组合，Helm 用于组件安装和可重复部署，GitHub 用于代码与 GitOps 状态托管。

从平台使用视角看，这套架构将应用接入动作收敛到 Backstage 表单和 GitHub PR 中，将部署、同步、构建和运行交给平台组件处理：

```text
服务接入方：填写 Backstage 模板 -> Review PR -> 合并
平台自动化组件：Tekton / Argo CD / Crossplane / Kubernetes 执行后续动作
平台维护侧：安装和维护底层组件、Secret、镜像仓库、网络入口
```

因此，Backstage 模板不是替代 Kubernetes，而是把 Kubernetes 平台能力包装成标准化的自服务入口。

## 3. 技术选型总览

| 技术 | 在项目中的角色 | 主要作用 | 选型理由 |
| --- | --- | --- | --- |
| Kubernetes | 平台运行底座 | 运行 Backstage、Tekton、Argo CD、Demo 服务等组件 | 云原生平台事实标准，适合作为 POC 展示底座 |
| kind | 本地/服务器测试集群 | 在服务器上快速创建 Kubernetes 集群 | 轻量、可复现、适合 POC 和演示环境 |
| Helm | 安装与打包工具 | 安装 Crossplane、Tekton、Argo CD、Backstage、Gateway 等 | 便于模块化部署和版本管理 |
| Crossplane | 平台资源抽象层 | 用声明式资源组合平台能力 | 体现平台工程的资源抽象与自服务能力 |
| Tekton | CI 引擎 | 接收触发事件，执行 clone、test、build、push、update-gitops 等任务 | Kubernetes 原生 CI，适合和集群内资源联动 |
| Argo CD | GitOps CD 引擎 | 监听 Git 仓库状态并同步 Kubernetes 资源 | 能清晰展示 GitOps 工作流 |
| Backstage | Dev Portal | 展示服务目录、服务入口、模板和平台能力 | 适合作为平台工程对开发者的统一入口 |
| Envoy Gateway | 服务入口网关 | 将 Demo 服务暴露到可访问端口 | Gateway API 生态，适合展示现代入口网关 |
| GitHub | 源码与状态仓库 | 存放应用代码、Catalog、GitOps 配置和触发 Webhook | 和 Backstage、Argo CD、Tekton 都容易集成 |
| GHCR / 镜像仓库 | 镜像分发 | 保存 Backstage、Demo 等镜像 | 和 GitHub 生态一致，适合 POC |
| Cloudflare Tunnel | Webhook 临时公网入口 | 让 GitHub Webhook 能访问内网 EventListener | 适合无公网 Ingress 的临时演示场景 |
| SSH Port Forward | 本机访问服务器服务 | 将服务器上的 Backstage、Demo、EventListener 暴露到本机 localhost | 简单稳定，适合调试和演示 |

## 4. Backstage：开发者门户

### 作用

Backstage 在本项目中承担 Dev Portal 的角色，是开发者看到平台能力的统一入口。

它主要负责：

- 展示服务目录，例如 `fastapi-demo-2`。
- 展示服务元数据，例如 owner、lifecycle、source repository。
- 提供 `Open Service` 链接跳转到实际运行的服务。
- 作为后续服务模板、自服务创建、平台文档入口的承载界面。

### 为什么选择 Backstage

Backstage 的优势是它不只是一个 Dashboard，而是围绕“软件目录”和“开发者体验”构建的门户。对于平台工程 POC 来说，它能把底层复杂的 Kubernetes、CI/CD、GitOps 能力包装成开发者容易理解和使用的入口。

### 当前项目中的关键文件

```text
crossplane-backstage-poc/catalog/services/fastapi-demo-2/catalog-info.yaml
crossplane-backstage-poc/apps/backstage-custom/app-config.production.yaml
crossplane-backstage-poc/manifests/backstage/backstage-values.yaml
```

其中 `catalog-info.yaml` 控制 Backstage Catalog 中服务卡片和 `Open Service` 链接。比如将服务入口从：

```text
http://localhost:30080/
```

改为：

```text
http://110.120.0.3:30080/
```

就是通过修改 Catalog entity 的 links，而不是修改 Helm 或 Kubernetes Service。

## 5. Crossplane：平台资源抽象层

### 作用

Crossplane 用于把底层 Kubernetes 资源、Helm Release、应用服务等能力抽象成更高层的平台资源。

它体现的是平台工程中的一个重要思想：

```text
平台团队定义抽象和组合
业务团队提交声明式资源
平台自动编排底层基础设施
```

### 为什么选择 Crossplane

如果只是部署几个 YAML，无法体现平台工程的抽象能力。Crossplane 可以通过 XRD、Composition、Claim 等机制，把复杂资源组合封装成平台 API。

### 在 POC 中的价值

- 展示平台资源声明式管理。
- 展示 Helm provider、Kubernetes provider 等 provider 能力。
- 为后续“服务自助创建”提供基础。

## 6. Tekton：CI 流水线

### 作用

Tekton 是本项目中的 CI 引擎，负责在代码变更或手动触发后执行构建流程。

典型流程：

```text
EventListener 接收事件
  -> TriggerBinding 解析参数
  -> TriggerTemplate 创建 PipelineRun
  -> Pipeline 执行多个 Task
  -> clone-repo
  -> test-fastapi-demo
  -> build-push-fastapi-demo
  -> update-fastapi-demo-gitops
```

### 为什么选择 Tekton

Tekton 是 Kubernetes 原生 CI/CD 工具，Pipeline、Task、PipelineRun 都是 Kubernetes CRD，便于和集群资源、RBAC、Secret、镜像构建、GitOps 更新联动。

### 触发方式

当前讨论过三种触发方式：

1. GitHub Webhook + Tunnel 自动触发。
2. 手动 `curl` EventListener 触发。
3. Git hook / 包装脚本在 push 后自动 curl 触发。

其中演示时最稳定的是：

```text
git push origin main && ./trigger-tekton.sh
```

或者直接手动 curl 触发 Tekton，避免 tunnel 和 GitHub Webhook 调试成本。

## 7. Argo CD：GitOps CD

### 作用

Argo CD 是 CD 层，负责把 Git 仓库中的目标状态同步到 Kubernetes 集群。

它的核心思想是：

```text
Git 仓库是期望状态
Kubernetes 集群是实际状态
Argo CD 负责持续对齐两者
```

### 为什么选择 Argo CD

Argo CD 的可视化和 GitOps 概念很适合演示。它能清楚说明：

- 谁修改了目标状态。
- 当前应用是否 Synced。
- 当前应用是否 Healthy。
- 集群内资源和 Git 仓库是否漂移。

### 注意事项

Argo CD 和 Helm 同时管理同一个 Kubernetes 资源时，容易发生所有权冲突。演示中应明确划分：

- Helm 管平台组件安装。
- Argo CD 管 GitOps 应用资源。
- 不要让 Helm 和 Argo CD 同时管理同一批 Tekton Task、Pipeline 或 AppService。

## 8. Helm：组件安装和部署包装

### 作用

Helm 在本项目中主要用于：

- 安装平台组件。
- 将平台模块拆成多个 chart。
- 管理 values 和环境差异。
- 让服务器复现部署流程更可控。

### 为什么不是一个大 Chart

一开始讨论过把 Tekton、Crossplane、Argo CD、Backstage 等全部打成一个大 Helm Chart。但更合理的方式是拆成多个 chart：

```text
platform-crossplane
platform-tekton
platform-argocd
platform-backstage
platform-gateway
platform-demo-appservices
```

原因：

- 每个组件生命周期不同。
- 出错时更容易定位。
- 可以按顺序安装和重试。
- 避免一个组件超时导致全量安装失败。
- 更符合平台组件分层。

## 9. Envoy Gateway：服务访问入口

### 作用

Envoy Gateway 负责把 Demo 服务暴露出来，让浏览器可以访问。

当前服务入口示例：

```text
http://110.120.0.3:30080/
```

或者在本机 SSH 转发后访问：

```text
http://localhost:30080/
```

### 为什么选择 Envoy Gateway

Envoy Gateway 基于 Gateway API，适合展示现代 Kubernetes 网关模型。相比只用 NodePort，它更能体现平台层的入口管理能力。

## 10. GitHub：源码、Catalog 和 GitOps 状态中心

### 作用

GitHub 在本项目中承担多种角色：

- 保存应用源码。
- 保存 Backstage Catalog 文件。
- 保存 GitOps 目标状态。
- 触发 Webhook。
- 保存镜像仓库 GHCR 的关联信息。

### 为什么选择 GitHub

GitHub 和 Backstage、Argo CD、Tekton、GHCR 都容易集成，适合 POC 快速完成闭环。

## 11. GHCR / 镜像仓库

### 作用

镜像仓库用于存放平台组件或业务服务镜像，例如：

```text
ghcr.io/re1lya/platform-poc-backstage:0.1.5
```

### 注意事项

如果镜像是私有的，服务器或集群节点需要登录或配置 imagePullSecret。如果网络较慢，可以使用：

- 先在本地拉取镜像，再 `docker save` 成 tar。
- 用 `rsync` 传到服务器。
- 在服务器 `docker load`。
- 对 kind 集群使用 `kind load docker-image`。

但镜像名和 tag 必须和 Pod 里声明的一致，否则 Kubernetes 仍会重新去远端拉取。

## 12. Cloudflare Tunnel：临时公网 Webhook 入口

### 作用

当服务器或 kind 集群没有公网入口时，GitHub Webhook 无法直接访问 Tekton EventListener。Cloudflare Tunnel 可以临时生成公网 URL，把请求转发到本机或服务器端口。

### 使用边界

Tunnel 只解决这段链路：

```text
GitHub Webhook -> Tunnel -> EventListener
```

如果使用手动 curl 触发 Tekton，则不需要 tunnel。

## 13. SSH Port Forward：调试和演示访问

### 作用

SSH 端口转发用于把服务器上的服务映射到本机。

例如 Backstage：

```bash
ssh -L 7007:127.0.0.1:7007 admin@服务器地址
```

Demo 服务：

```bash
ssh -L 30080:127.0.0.1:30080 admin@服务器地址
```

Webhook 调试：

```bash
ssh -L 18080:127.0.0.1:8080 admin@服务器地址
```

### 关键理解

SSH 不是让两边端口天然互通，而是让本机某个端口监听，并通过 SSH 转发到服务器能访问的地址和端口。

## 14. 创建应用服务的端到端链路

使用 Backstage 模板创建一个新服务时，完整链路应理解为：

```text
1. 开发者在 Backstage Create 页面选择模板
2. 填写服务名、代码仓库、镜像仓库、端口、namespace 等参数
3. Backstage Scaffolder 根据模板生成 GitOps 文件
4. Backstage 在 GitHub 上创建 Pull Request
5. 人工 Review PR，确认生成的 YAML 符合预期
6. PR 合并到 GitOps 仓库
7. Argo CD 发现新的 Application 或资源变更
8. Argo CD 同步 AppService Helm chart
9. Crossplane 处理 AppService 抽象资源
10. Kubernetes 中创建或更新 Deployment / Service 等运行时资源
11. Backstage Catalog provider 刷新后展示新服务
12. 后续代码 push 触发 Tekton CI
13. Tekton 构建镜像并更新 GitOps values 中的 image tag
14. Argo CD 再次同步，服务滚动更新
```

这里的关键点是：Backstage 负责“生成和提交配置”，Argo CD 负责“把 GitOps 状态同步到集群”，Crossplane 和 Helm 负责“把平台抽象转成 Kubernetes 资源”，Tekton 负责“后续代码变更后的构建和版本更新”。

## 15. 当前方案的边界

当前方案是 POC，不应夸大为生产级平台。

仍需增强的点：

- Webhook 入口需要稳定公网 Ingress 或正式域名，不应长期依赖临时 tunnel。
- 镜像拉取需要企业内镜像仓库或稳定镜像代理。
- Secret 管理需要接入正式密钥管理机制。
- Helm 与 Argo CD 的资源边界需要进一步固化。
- CI/CD 成功反馈需要更稳定的可观测脚本或 Dashboard。
- Backstage 模板、权限、审批和多租户能力仍需完善。
